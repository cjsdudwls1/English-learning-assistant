# 기술 문제 해결 4단계 프로세스

> 코드 에러, 버그, API 연동 오류 등 기술적 문제를 해결할 때 반드시 이 순서를 따른다.
> 감으로 코드를 수정하지 않는다.
> **기존에 잘 작동하던 다른 기능이나 디자인을 임의로 건드리지 않으며, 시스템의 정합성(Consistency)을 해치는 불필요한 수정을 절대 금한다.**

## 1단계: 현상 분석 및 근본 원인(Root Cause) 추론
- 공식 문서를 찾거나 코드를 수정하기 전에, 제공된 **에러 로그, 스택 트레이스, 현재 코드의 동작 맥락을 먼저 면밀히 분석**한다.
- 표면적인 오류 현상(예: 500 에러, 타입 에러 등)에 머물지 않고, **"왜 이 지점에서 문제가 발생했는지" 근본 원인에 대한 가설을 수립**한다.
- **주의:** 이 단계에서는 절대 해결 코드(수정안)를 먼저 제시하지 않으며, 원인 규명과 검증 타겟을 좁히는 데에만 집중한다.

## 2단계: 공식 문서 리스트화
- 1단계에서 세운 가설을 검증하고 문제의 원인을 정확히 파악하기 위해 **필요한 기술 공식 문서를 리스트로 정리**한다.
- 예시:
  - SDK 메서드 오류 → 해당 SDK 공식 문서
  - REST API 응답 구조 불일치 → API 레퍼런스 문서
  - 프레임워크 동작 이상 → 프레임워크 공식 가이드
- 프로젝트 내 관련 워크플로우 파일(예: `/veo-api`, `/backend`)도 함께 확인한다.

## 3단계: 공식 문서 낭독 및 필요 부분 채취
- 리스트에 정리한 공식 문서를 `read_url_content` 또는 `view_content_chunk`로 **직접 읽는다**.
- 문서에서 **현재 문제와 관련된 코드 패턴, API 응답 구조, 파라미터 사양** 등을 정확히 채취한다.
- 채취한 내용을 기반으로 현재 코드와 비교하여 **차이점(불일치)**을 명확히 식별한다.
- 추측이 아닌 **문서 근거로 1단계의 가설을 검증하고 최종 원인을 확정**한다.

## 4단계: 코드 수정
- 3단계에서 확인한 공식 사양에 맞게 코드를 수정한다.
- **[사이드 이펙트 차단] 오직 파악된 근본 원인을 해결하는 데 필요한 최소한의 코드만 타겟팅하여 수정한다. 문제 해결과 무관하게 기존에 잘 작동하던 다른 기능, 로직, UI/UX 디자인을 건드려 전체 시스템의 정합성을 훼손하는 짓은 절대 하지 않는다.**
- 수정 시 **왜 이렇게 바꾸는지**(공식 문서의 어떤 부분과 맞추는 것인지, 파악한 근본 원인이 어떻게 해결되는지)를 주석 또는 설명에 남긴다.
- 가능하면 디버그 로그를 추가하여 향후 유사 문제 발생 시 빠르게 진단할 수 있도록 한다.

## 적용 기준

이 프로세스는 다음 상황에서 **반드시** 적용한다:

- 런타임 에러 또는 타입 에러 발생 시
- 외부 API/SDK 연동 오류 시
- 응답 데이터 구조 불일치 시
- "왜 안 되는지" 원인이 즉시 명확하지 않은 경우

단순 오타, 변수명 오류 등 원인이 자명한 경우에는 1단계(원인 분석) 진행 후, 문서 검색(2~3단계)을 생략하고 바로 코드를 수정할 수 있다.
