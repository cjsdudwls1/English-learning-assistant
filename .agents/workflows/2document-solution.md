# 워크플로우 2/4: 공식 문서 기반 해결책 문서화

> 디버깅 파이프라인: `[1] 원인 분석` → **`[2] 해결책 문서화`** → `[3] 문서 검수` → `[4] 코드 수정`

**■ 개요 및 원칙**

* **목적:** 파악된 근본 원인을 바탕으로, 추측을 배제하고 **공식 문서에 근거한 명확한 해결책** 도출 및 문서화
* **수정 원칙:** 기존 기능 및 디자인 온전하게 보존, 시스템 정합성 유지, 필수 부분만 최소한으로 수정

**■ 전제 조건**

* `/1analyze-root-cause`를 통해 작성된 원인 분석 문서(`.docs` 폴더)가 존재할 것

**■ 진행 절차**

**1단계: 원인 문서 확인**

* **원인 파악:** 사용자가 언급한 문서 내 명시된 '원인' 정확히 숙지
* **문제 정의:** 원인 문서의 가설 및 근거 재검토를 통한 해결 대상(핵심 문제) 명확화

**2단계: 관련 공식 문서 탐색 및 직접 확인**

* **목록화:** 문제 해결에 필요한 기술 문서(SDK, API 레퍼런스, 프레임워크 가이드 등) 리스트업
* **직접 열람:** `read_url_content` 또는 `view_content_chunk`를 활용해 문서 내용 직접 확인
* **내부 검토:** 필요시 프로젝트 내 관련 워크플로우 파일(`/veo-api`, `/backend` 등) 함께 확인

**3단계: 관련 코드 패턴 및 사양 채취**

* **정보 수집:** 공식 문서 내 현재 문제와 연관된 코드 패턴, API 응답 구조, 파라미터 사양 등 추출
* **근거 확보:** 수집한 사양 및 공식 예제 코드를 해결책 도출의 핵심 근거로 정리

**4단계: 현재 코드와 비교 및 차이점 식별**

* **대조 분석:** 수집한 공식 사양과 현재 프로젝트 코드 비교
* **문제 구체화:** 불일치하는 부분 식별 및 해당 차이가 문제를 일으키는 이유 구체적으로 명시

**5단계: 해결책 도출 및 `.docs` 문서 추가 작성**

* **해결책 마련:** 식별된 차이점과 공식 문서를 근거로 명확한 해결 방안 도출
* **문서 업데이트:** 기존 `.docs` 폴더 내 원인 분석 문서에 도출된 해결책 추가
* **필수 포함 항목:**
  * **참고 공식 문서:** URL 및 해당 섹션 명시
  * **차이점 분석:** 공식 사양과 현재 코드의 구체적인 비교 내역
  * **해결책:** 구체적인 수정 방법 및 기반이 된 공식 근거
  * **예상 영향 범위:** 수정 시 파급될 수 있는 타 모듈 및 기능 영향도

**■ 주의사항**

* **직접 확인 원칙:** 공식 문서를 직접 읽고 숙지한 상태에서만 해결책 제안
* **팩트 기반:** 문서에서 직접 확인하여 검증된 내용만 근거로 활용
* **최소 변경:** 시스템 안정성을 위해 문제를 해결할 수 있는 최소한의 변경만 지향
