# 이미지 분석 파이프라인 변경 규칙

> **핵심 원칙**: Cloud Function은 Edge Function(커밋 `b6fd71be`)의 **로직과 프롬프트를 그대로 이식**한 것이다.
> 플랫폼만 Deno Edge Function → Node.js GCP Cloud Function으로 바뀌었을 뿐, 분석 로직 자체는 동일해야 한다.

---

## 원본 참조 코드 위치

```
.agents/reference/edge-function-b6fd71be/
├── index.ts              # 메인 오케스트레이터 (배치 병렬, 메모리 해제, waitUntil)
├── prompts.ts            # 모든 프롬프트 원본 (Pass A/0/B/C)
├── pageAnalyzer.ts       # 4-Pass 분석 흐름 + mergeHandwritingMarks
├── analysisProcessor.ts  # 모델 failover + 분류
├── imageCropper.ts       # WASM 뮤텍스 + 듀얼 크롭
├── sessionManager.ts     # 세션 생성/완료 + Vertex AI 인증 검증
├── labelProcessor.ts     # taxonomy 보강 + O/X 마크 정답 판정
├── problemSaver.ts       # content JSONB 구조 + choices 정규화
├── validation.ts         # normalizeMark, cleanOrNull, makeDepthKey
├── problemProcessor.ts   # 문제 처리 유틸
├── imageUploader.ts      # Storage 업로드
├── requestParser.ts      # 요청 파싱 + MAX_IMAGES 제한
├── taxonomyLoader.ts     # taxonomy depth↔code 양방향 Map
└── metadataGenerator.ts  # 메타데이터 생성
```

## 작업 전 필수 체크리스트

Cloud Function의 `cloud-functions/analyze-image/` 아래 파일을 **수정하기 전에** 반드시 다음을 수행한다:

### 1단계: 원본 코드 읽기
// turbo
```
view_file .agents/reference/edge-function-b6fd71be/<대응 파일>
```
Cloud Function에서 수정하려는 파일에 대응하는 Edge Function 원본 파일을 먼저 읽는다.

**파일 매핑표:**

| Cloud Function 파일 | Edge Function 원본 파일 |
|---------------------|------------------------|
| `index.js` | `index.ts` |
| `shared/prompts.js` | `prompts.ts` |
| `shared/passes.js` | `pageAnalyzer.ts` + `analysisProcessor.ts` |
| `shared/imageCropper.js` | `imageCropper.ts` |
| `shared/dbOperations.js` | `sessionManager.ts` + `labelProcessor.ts` + `problemSaver.ts` + `imageUploader.ts` |
| `shared/aiClient.js` | (신규, 대응 없음 - Edge에서는 `_shared/aiClient.ts` 공용 모듈) |
| `shared/errors.js` | (Edge `_shared/errors.ts`와 동일 구조) |
| `shared/taxonomy.js` | `taxonomyLoader.ts` |
| `shared/config.js` | (신규, Edge에서는 `_shared/models.ts`) |
| `shared/imagePreprocessor.js` | (Cloud Function에만 존재 — 유지) |

### 2단계: 동등성 검증 질문

수정 전, 다음 질문에 모두 "예"로 답할 수 있어야 한다:

1. **프롬프트 동등성**: 수정 후에도 Edge Function의 프롬프트 텍스트와 동일한가?
   - `prompts.ts`의 `buildStructurePrompt`, `buildBoundingBoxPrompt`, `buildCroppedUserAnswerPrompt`, `buildCroppedCorrectAnswerPrompt`, `buildHandwritingDetectionPrompt`, `buildClassificationPrompt` 원문과 비교

2. **파이프라인 동등성**: Pass A+0 병렬 → 크롭 → Pass B(user+correct 병렬) → Pass C 흐름이 유지되는가?
   - `pageAnalyzer.ts`의 `analyzeOnePage` 흐름과 비교

3. **데이터 구조 동등성**: DB 저장 구조(content JSONB, labels classification)가 동일한가?
   - `problemSaver.ts`의 `buildContentJson`과 비교
   - `labelProcessor.ts`의 `buildLabelsPayload`와 비교

4. **검증 로직 동등성**: 답안 범위 검증, O/X 마크 처리, taxonomy 보강이 유지되는가?
   - `pageAnalyzer.ts`의 `mergeHandwritingMarks`와 비교
   - `labelProcessor.ts`의 is_correct 판정 로직과 비교

### 3단계: 변경 후 확인

수정 완료 후, 다음을 확인한다:

1. `grep_search`로 Edge Function에만 있고 Cloud Function에 없는 핵심 키워드 확인:
   - `user_marked_correctness`
   - `taxonomyByDepthKey`
   - `visual_context`
   - `shared_passage_ref`
   - `question_body`

2. 난이도 정규화가 영어/한국어 양방향으로 동작하는지 확인

3. 세션 완료 시 `.eq('status', 'processing')` 가드가 있는지 확인

## 절대 금지 사항

1. **프롬프트 임의 수정 금지**: Edge Function 원본 프롬프트를 사용자 명시적 요청 없이 변경하지 않는다. 프롬프트 변경은 분석 정확도에 직접적 영향을 미친다.

2. **로직 단순화 금지**: "Cloud Function이니까 간단하게" 식의 로직 생략/단순화를 하지 않는다. Edge Function의 모든 검증/보강/폴백 로직은 의도적으로 설계된 것이다.

3. **신규 로직 추가 시 원본 확인**: 새로운 기능을 추가할 때, Edge Function에 해당 기능이 어떻게 구현되어 있었는지 먼저 확인하고, 있다면 그 방식을 따른다.

## 현재 알려진 누락 사항 (복원 대상)

아래 항목들은 이관 시 누락된 것으로, 복원이 필요한 로직이다:

| 심각도 | 누락 항목 | 원본 참조 파일 |
|--------|-----------|---------------|
| CRITICAL | Taxonomy depth→code/CEFR/난이도 보강 | `labelProcessor.ts` |
| CRITICAL | O/X 채점 마크 기반 정답 판정 | `labelProcessor.ts` |
| CRITICAL | 필기 답안 범위 검증 (1~5) | `pageAnalyzer.ts` |
| HIGH | 세션 완료 labeled 상태 가드 | `sessionManager.ts` |
| HIGH | content JSONB 상세 필드 | `problemSaver.ts` |
| HIGH | 배치 병렬 처리 (batch_size=3) | `index.ts` |
| HIGH | Labels 저장 실패 시 StageError | `index.ts` |
| MEDIUM | 토큰 사용량 DB 로깅 | `index.ts` |
| MEDIUM | 사용자 프로필 언어 자동 감지 | `index.ts` |
| MEDIUM | 난이도 양방향 정규화 | `index.ts` |
| MEDIUM | image_urls 검증/정리 | `sessionManager.ts` |
| MEDIUM | Pass C visual_context 이미지 전달 | `pageAnalyzer.ts` |
