# 다른 AI에게 진단 요청용 질문 생성 워크플로우

프로젝트에서 해결되지 않는 문제가 있을 때, 프로젝트 맥락을 전혀 모르는 외부 AI에게 질문할 수 있는 **자기완결적(Self-contained) 진단 요청 문서**를 생성한다.

## 핵심 원칙

- 외부 AI는 우리 프로젝트의 코드, 구조, 히스토리를 **전혀 모른다**.
- 따라서 문서 하나만 읽으면 문제를 완전히 이해하고 답변할 수 있어야 한다.
- 코드 조각, 프롬프트 전문, 데이터 흐름을 **생략 없이** 포함한다.

## 절차

### 1. 문제 범위 파악
- 사용자가 설명하는 **증상**을 정확히 기록한다.
- 관련된 파일들을 식별하고 읽는다:
  - 프롬프트 파일 (예: `prompts.ts`)
  - 처리 로직 파일 (예: `analysisProcessor.ts`, `index.ts`)
  - 검증/후처리 파일 (예: `validation.ts`, `labelProcessor.ts`)
  - 모델 설정 파일 (예: `models.ts`)

### 2. 문서 구조 (반드시 이 순서를 따른다)

```markdown
# [문제 제목] - 진단 요청

## 1. 프로젝트 개요
- 프로젝트가 무엇을 하는지 1~2문장으로 설명
- 사용 기술 스택 (Gemini API, Supabase Edge Function 등)

## 2. 아키텍처 / 데이터 흐름
- 전체 파이프라인을 단계별로 설명
- 각 단계에서 사용하는 **프롬프트 전문**을 코드블록으로 포함
- 단계 간 데이터가 어떻게 전달되는지 설명
- 관련 코드 조각(병합 로직, 검증 로직 등)을 포함

## 3. 입력 데이터 설명
- 테스트에 사용한 입력 데이터의 구체적 특성
- 이미지라면: 레이아웃, 내용물, 마크 위치 등을 텍스트로 상세 서술
- (가능하면 이미지 자체를 첨부하라고 사용자에게 안내)

## 4. 기대 결과 vs 실제 결과
- 각 항목별로 기대한 값과 실제 나온 값을 표 또는 리스트로 비교
- 패턴이 있는지 명시 (예: "3개 중 1개만 성공", "이전에 되던 것이 안됨")

## 5. 이전 시도 이력
- 시도 1: 무엇을 했고, 결과가 어땠는지
- 시도 2: 무엇을 했고, 결과가 어땠는지
- 각 시도에서 왜 실패했다고 판단하는지

## 6. 구체적 질문 (번호 매기기)
- 근본 원인이 무엇이라고 생각하는가?
- 프롬프트의 어떤 부분이 문제인가?
- 어떻게 수정해야 하는가?
- 대안적 접근이 있는가?
```

### 3. 작성 규칙

- **프롬프트는 반드시 전문을 포함한다.** "위에서 설명한 프롬프트"와 같은 참조를 쓰지 않는다.
- **코드는 핵심 로직만 발췌**하되, 생략 시 `// ... (무관한 로직 생략)` 주석을 단다.
- **모델명을 구체적으로 명시**한다 (예: `gemini-2.5-flash`, `gemini-3.1-flash-lite-preview`).
- **한국어와 영어를 혼용**해도 된다. 프롬프트 원문은 영어 그대로 유지.
- **추측/가설이 있다면** "우리는 이것이 원인이라고 의심하지만 확실하지 않다"라고 솔직히 쓴다.

### 4. 결과물 저장
- artifacts 디렉토리에 저장한다.
- 파일명: `question_for_other_ai.md`
- 사용자에게 "이 문서를 복사해서 다른 AI에게 전달하세요. 테스트 이미지도 함께 첨부하면 더 정확한 답을 받을 수 있습니다."라고 안내한다.
