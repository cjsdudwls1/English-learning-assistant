# 워크플로우 4/4: 프로젝트 맞춤형 안전한 코드 수정

> 디버깅 파이프라인: `[1] 원인 분석` → `[2] 해결책 문서화` → `[3] 문서 검수` → **`[4] 코드 수정`**

**■ 개요 및 원칙**

* **목적:** 검수 완료된 문서의 해결책을 바탕으로, 프로젝트 정합성과 안정성을 확보하며 코드를 수정
* **수정 원칙:** 기존 정상 작동 기능 및 디자인을 온전히 보존하며, 시스템 정합성(Consistency) 유지 범위 내 필수 부분만 최소한으로 수정

**■ 전제 조건**

* `/3verify-docs`를 통해 검수 완료된 해결책 문서(`.docs` 폴더)가 존재할 것

**■ 진행 절차**

**1단계: 해결책 문서 확인**

* 사용자 언급 또는 검수 완료된 `.docs` 문서의 **'해결책'** 기반 코드 수정
* 해결책의 핵심 요지 및 공식 문서 근거 재확인

**2단계: 프로젝트 아키텍처 및 코딩 컨벤션 파악**

* 수정 대상 코드 주변의 기존 **아키텍처 패턴** 분석
* 프로젝트 **코딩 컨벤션**(네이밍, 폴더 구조, 임포트 방식, 에러 처리 패턴 등) 파악
* 관련 유틸리티, 헬퍼, 상수 등 **재사용 가능한 기존 코드** 파악 및 활용 검토

**3단계: 수정 범위 최소화 및 타겟팅**

* **사이드 이펙트 방지:** 근본 원인 해결에 필요한 최소한의 코드만 타겟팅하여 집중 수정
* **정합성 유지:** 기존 정상 작동 기능, 로직, UI/UX 디자인 철저히 보존
* **영향도 파악:** 수정 전 영향받을 수 있는 타 모듈 및 컴포넌트 사전 식별

**4단계: 코드 수정 및 정합성 유지**

* 공식 사양 및 해결책 원리를 프로젝트 기존 아키텍처·컨벤션에 맞게 **자연스럽게 반영**
* 공식 문서 예제 코드 단순 복사 지양, 프로젝트 스타일에 맞게 **적용(Adapt)**

**5단계: 수정 근거 및 디버그 로그 추가**

* **주석 작성:** 수정 사유(공식 문서 근거, 원인 해결 방식 등) 명확히 기재
* **디버그 로그 도입:** 향후 유사 문제 진단을 위해 적극 추가 고려
  * 프로덕션 환경에서 비활성화 가능한 조건부 로깅 적용
  * 에러 맥락 정보(입력값, 상태, 타임스탬프 등) 포함
  * 기존 로깅 패턴과 일관된 형식 유지

**6단계: 수정 결과 검증**

* **타입 및 빌드 검증:** `npx tsc --noEmit` 등 타입 체크 및 빌드 수행
* **연관 기능 검증:** 수정 부분 및 관련된 타 기능의 정상 동작 여부 교차 확인
* **실제 동작 검증:** 브라우저 또는 터미널 테스트를 통해 문제 해결 최종 확인

**■ 주의사항**

* **최우선 원칙:** 기존 정상 작동 기능의 안정적 유지 및 보존
* **수정 범위 엄수:** 문서 명시 범위 내로 수정 한정 (추가 수정 필요 시 **문서 선행 업데이트**)
* **사전 협의:** 예상보다 수정 범위 확대 시 사용자와 사전 확인 필수
* **점진적 수정:** 일괄 대규모 수정 지양, 작은 단위로 분할하여 점진적 진행 권장

**■ 분기 조건**

* **검증 실패 시:** 수정 결과가 기대와 다르거나 새로운 문제가 발생한 경우 → `/1analyze-root-cause`로 돌아가 새로운 원인 분석 수행
