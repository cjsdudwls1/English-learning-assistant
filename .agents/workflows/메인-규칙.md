# English Learning Assistant — AI Editor Main Rules

Tech Stack: React (Vite), TypeScript, Supabase (Database, Auth, Edge Functions, Storage), Google Vertex AI (Gemini), Netlify.

---

## Absolute Rules (Core Directives)

1. **Maintainability First:** All code must be clean, readable, and structured for safe modification 6 months later. No throwaway "just make it work" code. Follow the quality checklist in `/완벽한-디버깅` Phase 4 for every code change.

2. **Complete Deletion (No Traces):** When removing a feature, trace and remove ALL references — imports, state variables, useEffect hooks, UI components, type definitions. Never hide code with `{false && (`. Use `grep_search` to find every related keyword.

3. **Security Mandate:** Environment variables must only be accessed server-side (Edge Functions). Never expose API keys to the browser.

4. **Serverless Constraints:** Always consider Supabase Edge Function (Deno) timeout and memory limits. Synchronous wait loops (`while + sleep`) are forbidden. Use async polling/retry patterns.

5. **Integrity Verification:** After code changes, always run `npx tsc --noEmit` and confirm zero compilation errors. Never declare "done" before compilation succeeds.

6. **Minimal Change Scope:** Only modify what was requested. Do not touch unrelated files. Refactoring is performed only when explicitly requested.

7. **Search Before Creating:** Before writing new functions/utilities, use `grep_search` to check if similar functionality already exists. Duplicate code creation is forbidden.

8. **Pre-Commit Quality Gate:** After completing any implementation or code modification, run the quality checklist from `/완벽한-디버깅` Phase 4-5 before declaring "done." Specifically:
   - Structural rules (function ≤40 lines, file ≤200 lines, nesting ≤3 levels, params ≤3)
   - Naming rules (intent-revealing names, boolean prefixes, verb-first functions, UPPER_SNAKE_CASE constants)
   - No duplication (DRY, reuse existing utilities)
   - Error handling (no empty catch, contextual error messages)
   - Type safety (no `any`, minimal `as`, explicit return types)
   - Dependency rules (no circular deps, clean imports)

---

## Refactoring Principles

When refactoring is explicitly requested, follow these principles:

1. **I/O Contract Integrity:** Define TypeScript interfaces for all module inputs/outputs. Verify that existing callers continue to work without modification. If caller changes are unavoidable, document the impact scope and use re-export or adapter patterns.

2. **Official Documentation Basis:** Verify that proposed patterns are recommended in the latest official docs for the relevant technology stack. Use `read_url_content`, `context7`, or `search_web` to directly confirm.

3. **Incremental Milestones:** Split implementation into independently verifiable steps. Each step must pass build/type checks before proceeding to the next.

4. **Design Before Code:** For large-scale refactoring, create an `implementation_plan.md` artifact documenting architecture, folder structure, I/O contracts, and milestones before writing any code.
