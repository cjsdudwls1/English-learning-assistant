# 워크플로우: 코드 자가 품질 검증 (Self-Review)

> **적용 시점:** 기능 구현 또는 코드 수정이 **완료된 직후**, 커밋 또는 배포 **직전**에 실행한다.
> **핵심 목표:** 작성한 코드가 `/7code-quality` 규칙을 준수하며, 6개월 후에도 안전하게 유지보수할 수 있는 상태인지 검증한다.

---

## 1단계: 정적 분석 (Static Analysis)

작성한 코드가 빌드/타입 체크를 통과하는지 확인한다.

* **TypeScript 타입 체크:** `npx tsc --noEmit` 실행, 에러 0건 확인
* **빌드 검증:** `npm run build` 실행, 빌드 성공 확인
* **린트 경고 확인:** 새로 발생한 린트 경고가 있다면 해결

> 1단계를 통과하지 못하면 이후 단계로 진행하지 않는다.

---

## 2단계: 유지보수성 체크리스트 (Maintainability Checklist)

작성한 코드를 아래 체크리스트에 대조하여 검증한다. **모든 항목에 대해 판정 (통과/위반)을 기록한다.**

### 구조 검증
- [ ] 40줄을 초과하는 함수가 있는가? → 있다면 분리
- [ ] 200줄을 초과하는 파일이 있는가? → 있다면 분리
- [ ] 3단계를 초과하는 중첩(if/for/while)이 있는가? → 있다면 평탄화
- [ ] 매개변수가 4개 이상인 함수가 있는가? → 있다면 옵션 객체 패턴 적용

### 네이밍 검증
- [ ] 의도가 불명확한 변수명이 있는가? (x, data, temp, res, val, item 등)
- [ ] 불린 변수에 is/has/should/can 접두사가 빠져 있는가?
- [ ] 함수명이 동사로 시작하지 않는 경우가 있는가?

### 중복 검증
- [ ] 동일하거나 유사한 로직이 2곳 이상 반복되는가? → `grep_search`로 확인
- [ ] 매직 넘버/스트링이 상수화 없이 직접 사용되었는가?
- [ ] 기존 유틸리티로 대체 가능한 새 함수를 만들지는 않았는가?

### 타입 안전성 검증
- [ ] `any` 타입을 사용한 곳이 있는가?
- [ ] 불필요한 타입 단언(`as`)을 사용한 곳이 있는가?
- [ ] export 함수에 반환 타입이 누락되었는가?

### 에러 처리 검증
- [ ] 빈 catch 블록이 있는가?
- [ ] 에러 로그에 [함수명] 접두사와 맥락 정보가 포함되어 있는가?
- [ ] 사용자 대면 에러에 기술적 메시지가 그대로 노출되지는 않는가?

### 의존성 검증
- [ ] 순환 참조(Circular Dependency)가 발생하지 않는가?
- [ ] 사용하지 않는 import가 남아있지 않는가?

## 3단계: "6개월 후 테스트" (Future Developer Test)

코드를 처음 보는 개발자의 시점에서 다음 질문에 답한다:

1. **이 파일의 역할을 파일명과 첫 10줄만 보고 파악할 수 있는가?**
   - 파악할 수 없다면: 파일명 변경 또는 상단에 모듈 설명 주석 추가

2. **각 함수가 무엇을 하는지 함수명과 시그니처만 보고 알 수 있는가?**
   - 알 수 없다면: 함수명 개선 또는 JSDoc 주석 추가

3. **비즈니스 로직의 "왜"가 코드나 주석으로 설명되어 있는가?**
   - 설명되어 있지 않다면: Why 주석 추가

4. **이 코드를 수정해야 한다면, 어디를 고쳐야 하는지 즉시 찾을 수 있는가?**
   - 찾기 어렵다면: 모듈 분리 또는 네이밍 개선 필요

5. **이 코드를 수정했을 때, 다른 곳이 깨질 위험이 명확히 보이는가?**
   - 보이지 않는다면: 결합도(coupling)가 높은 것. 인터페이스 경계 재정의 필요

---

## 4단계: 변경 영향 최종 확인

* **관련 기능 수동 테스트:** 변경한 코드와 관련된 기존 기능이 정상 동작하는지 확인
* **브라우저/런타임 검증:** 실제 환경에서 변경 사항이 예상대로 작동하는지 확인
* **콘솔 에러 확인:** 브라우저 콘솔 또는 서버 로그에 새로운 에러/경고가 없는지 확인

---

## 5단계: 변경 요약 기록

최종 검증을 마친 후, 변경 사항을 간단히 요약한다:

* **변경한 파일 목록**
* **각 파일에서 변경한 내용 1줄 요약**
* **위반 사항 발견 및 조치 내역** (있었다면)
* **남은 기술 부채 또는 TODO** (있다면)

## 분기 조건

* **체크리스트 위반 발견 시:** 즉시 해당 코드를 수정하고, 수정 후 1단계부터 재검증
* **구조적 문제 발견 시:** 규모가 크다면 `/6plan-refactoring`으로 분기하여 설계도 먼저 작성
* **모든 검증 통과 시:** 커밋/배포 진행
