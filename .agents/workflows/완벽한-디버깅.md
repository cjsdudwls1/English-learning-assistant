# Complete Debugging Workflow

> **Purpose:** A debugging process that identifies root causes based on official documentation evidence, while simultaneously guaranteeing code quality and maintainability of all fixes.
> **Core Principle:** The approach of "just make it work and fix it later" is **strictly forbidden**. Never create new technical debt while fixing bugs.
> **Side Effect Prevention:** Never arbitrarily modify other features or designs that are already working correctly. Any unnecessary changes that compromise system consistency are strictly forbidden.

---

## Phase 0: Reconnaissance — Assess Current Code State

Before touching any code, **survey the terrain (codebase)**.

### 0-1. Explore Related Code
- Use `grep_search`, `find_by_name`, `list_dir` to quickly identify files/functions related to the issue.
- Map not only the file where the error occurred, but also the **modules it depends on and modules that depend on it**.

### 0-2. Identify Existing Quality Debt
- Quickly scan whether the code in the problem area already violates quality rules:
  - Functions exceeding 40 lines
  - Files exceeding 200 lines
  - Nesting deeper than 3 levels
  - Usage of `any` type
  - Empty `catch` blocks
- Existing violations may be **directly related to the current bug**, so this check is mandatory.

### 0-3. Check for Existing Utilities and Similar Code
- Check `utils/`, `lib/`, `helpers/`, `shared/` for reusable utilities applicable to the fix.
- Use `grep_search` to find patterns in the codebase where the same problem was previously solved.

---

## Phase 1: Symptom Analysis and Root Cause Hypothesis

> **Goal of this phase:** Never propose fix code first. Focus exclusively on identifying the cause.

### 1-1. Precise Error Analysis
- Read error logs, stack traces, and console output **line by line without skipping**.
- Go beyond surface-level symptoms (500 errors, TypeError, etc.) and trace the **causal chain of "why it happened at this specific point"**.

### 1-2. Formulate Root Cause Hypotheses
- Formulate at least **2-3 hypotheses**.
- For each hypothesis, specify the **verification method** (which logs to check, which values to print).

### 1-3. Prioritize Hypotheses
- Rank hypotheses from most to least likely, with supporting evidence:
  ```
  Hypothesis 1 (High probability): [Content] — Evidence: [Which part of error log]
  Hypothesis 2 (Medium probability): [Content] — Evidence: [Which code flow]
  Hypothesis 3 (Low probability): [Content] — Evidence: [Possible but insufficient evidence]
  ```

### 1-4. Obvious Bug Exception
- For trivially obvious causes (simple typos, import path errors, etc.):
  - Formulate only 1 hypothesis and explicitly state **why it's obvious**.
  - Skip Phases 2-3 and proceed directly to Phase 4.

---

## Phase 2: Official Documentation Listing

### 2-1. Create Documentation Checklist
- List the **specific official documents** needed to verify Phase 1 hypotheses:
  | Hypothesis | Required Document | Verification Point |
  |------------|-------------------|-------------------|
  | SDK method signature change | SDK official docs (per version) | Method parameters, return types |
  | API response structure mismatch | REST API reference | Response JSON schema |
  | Framework behavior change | Framework migration guide | Breaking changes list |

### 2-2. Include Project-Internal Documents
- Include project workflow files, README, configuration files in the review scope.
- Check whether Knowledge Items (KI) contain relevant information.

---

## Phase 3: Documentation Review and Hypothesis Verification

### 3-1. Directly Read Documentation
- Use `read_url_content`, `view_content_chunk`, `mcp_context7_query-docs`, etc. to **directly read** documents from the list.
- Extract evidence from **documentation originals**, not guesses.

### 3-2. Compare Current Code Against Documentation
- Perform a **1:1 comparison** between documented correct patterns/specifications and current code.
- Clearly identify and record discrepancies:
  ```
  [Discrepancy #1]
  - Documentation spec: fetchUser() returns Promise<User | null>
  - Current code: Return value asserted as User (as User), null case unhandled
  - Related hypothesis: Hypothesis 1
  ```

### 3-3. Confirm Final Root Cause
- Based on documentation evidence, **confirm the definitive root cause** from Phase 1 hypotheses.
- Clearly state the evidence for this confirmation.

---

## Phase 4: Code Fix (With Simultaneous Quality Rule Enforcement)

> **Key:** While fixing the bug, **simultaneously** verify that the fix code complies with quality rules.
> Target **only the minimum code necessary** to resolve the identified root cause.

### 4-1. Define Fix Scope
- **Explicitly list** files and functions to be modified.
- For each modification, attach **justification** (which discrepancy it resolves).
- **Never touch code outside the fix scope.**

### 4-2. Execute Fix + Quality Rule Checklist

Verify that the code being modified **complies with ALL rules below** while writing:

#### [Structural Rules]
- [ ] Are modified/added functions **40 lines or fewer**?
- [ ] Are modified files **200 lines or fewer**?
- [ ] Does each modified function do **only one thing**? (Single Responsibility)
- [ ] Is conditional/loop nesting **3 levels or fewer**?
- [ ] Do functions have **3 or fewer parameters**? (Use Options Object pattern if exceeding)

#### [Naming Rules]
- [ ] Can the **role and intent be immediately understood** from variable/function names alone?
- [ ] Are ambiguous names like `data`, `temp`, `res`, `val`, `item` avoided?
- [ ] Do boolean variables use `is`, `has`, `should`, `can` prefixes?
- [ ] Do function names **start with a verb**?
- [ ] Are magic numbers/strings extracted as **UPPER_SNAKE_CASE constants**?

#### [No Duplication (DRY)]
- [ ] Does the fix code **not duplicate** existing logic elsewhere?
- [ ] Are there existing utilities that could be reused? (Leverage Phase 0 findings)

#### [Error Handling]
- [ ] Are there **no empty `catch` blocks**?
- [ ] Do error logs include **[FunctionName/ModuleName] prefix + context info**?
- [ ] Are **technical error messages not exposed directly** to end users?

#### [Type Safety]
- [ ] Is `any` type **not used**?
- [ ] If `as` type assertion is used, is the **reason documented in a comment**?
- [ ] Are `export` function return types **explicitly declared**?

#### [Dependency Rules]
- [ ] Does the fix **not create circular dependencies**?
- [ ] Are there **no unused imports** added?

#### [Comments/Documentation]
- [ ] Is there documentation of **why this fix is needed** (official doc evidence, how root cause is resolved) in comments or commit messages?
- [ ] Are non-obvious business logic sections accompanied by context comments?

#### [Testability]
- [ ] Has logic that **can be pure functions** been separated as such?
- [ ] Are external service calls **not hardcoded**?

### 4-3. Add Debug Logging
- Add **debug logs at strategic locations** to enable rapid diagnosis of similar future issues.
- Log format: `console.error('[ModuleName:FunctionName] Context description:', variable);`

---

## Phase 5: Post-Fix Verification

### 5-1. Functional Verification
- Confirm that the fix **resolves the original error**.
- If possible, reproduce the same scenario and verify the **error no longer occurs**.

### 5-2. Side Effect Verification
- Confirm the fix **does not affect other previously working features**.
- If other modules import the modified file, verify those modules still function correctly.

### 5-3. Final Quality Check
- **Review the Phase 4 quality checklist one more time** for any missed items.
- Specifically re-verify:
  - Does the modification cause any file to exceed 200 lines?
  - Does the modification cause any function to exceed 40 lines?
  - Were any new `any` types or empty `catch` blocks introduced?

### 5-4. Fix Summary Report
- Summarize the final results in the following format:
  ```
  ## Debugging Result Report

  ### Root Cause
  [One-line summary of confirmed root cause]

  ### Documentation Evidence
  [Referenced official documents and relevant specifications]

  ### Changes Made
  - [Filename]: [Summary of changes]
  - [Filename]: [Summary of changes]

  ### Quality Rule Compliance
  - Structural rules: Pass/Fail (reason)
  - Naming rules: Pass/Fail (reason)
  - Error handling: Pass/Fail (reason)
  - Type safety: Pass/Fail (reason)

  ### Residual Risks
  [Specify if any, otherwise "None"]
  ```

---

## Application Criteria

This process **must** be applied in the following situations:

- Runtime errors or type errors
- External API/SDK integration failures
- Response data structure mismatches
- Cases where "why it doesn't work" is not immediately clear
- Cases where existing code quality issues may have caused the bug

## Exception (Fast Path)

When the cause is **completely obvious** (simple typo, import path error, etc.):
1. Perform **Phase 0** at minimum scope
2. In **Phase 1**, formulate only 1 hypothesis and state evidence of its obviousness
3. **Skip Phases 2-3**
4. In **Phase 4**, the quality checklist is **still mandatory**
5. **Phase 5** verification is **still mandatory**

---

## Absolute Prohibitions

1. **No gut-feeling fixes** — Never modify code based on "I think this might be the problem" without evidence.
2. **No shotgun fixes** — Never change multiple things simultaneously hoping "one of these will work."
3. **No "just make it work" fixes** — Temporary fixes that ignore quality rules are forbidden.
4. **No scope creep** — Never touch code unrelated to the problem resolution.
5. **No new technical debt** — Never introduce `any`, empty `catch` blocks, magic numbers, or other quality violations while fixing bugs.
