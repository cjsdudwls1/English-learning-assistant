# 워크플로우 4: 수정 전 영향도 사전 점검 (Pre-change Impact Analysis)

> 디버깅 파이프라인: `[1] 원인 분석` → `[2] 해결책 문서화` → `[3] 문서 검수` → **`[4] 영향도 사전 점검`** → `[5] 코드 수정`

**전제 조건**

* 해결책 문서(solution doc)가 완성되어 있을 것
* 해결책 문서의 팩트 체크(`/3verify-docs`)가 통과된 상태일 것
* 수정 **예정** 함수·모듈·파일이 명확히 식별된 상태일 것

**목적**

* 해결책 문서에 명시된 변경 사항을 적용**할 경우**, 기존 정상 작동 기능·로직에 악영향(사이드 이펙트)이 발생할 가능성이 없는지 **코드를 수정하기 전에** 사전 점검
* "수정해도 안전하다"는 확신을 코드 근거와 함께 확보한 뒤에만 실제 수정 단계로 진행

**핵심 원칙:** "증거 기반 사전 판단" — 추측이 아닌 코드 검색·타입 분석·호출 체인 추적으로 안전성을 **미리** 입증

---

**1단계: 변경 예정 범위 정리**

* 해결책 문서에서 **변경될 함수 시그니처, 로직, 파라미터** 목록 추출
* 변경 유형 분류:
  * 시그니처 변경 (매개변수 추가/제거/타입 변경)
  * 내부 로직 변경 (분기, 반환값, 에러 처리)
  * import/export 경로 변경
* 각 변경 항목에 대해 **하위 호환성이 유지될 것인지** 사전 판단 (기본값, optional 여부 등)

**2단계: 호출 체인 전수 조사**

* `grep_search`로 수정 예정 함수·모듈의 **모든 호출 위치** 식별
* 단순 import만 하고 해당 함수를 사용하지 않는 파일은 제외
* 각 호출 위치에 대해 다음을 확인:
  * 현재 호출 방식 (인자 수, 인자 타입)
  * 변경 적용 후에도 **기존 호출이 그대로 동작할 것인지** (기본 매개변수, 타입 호환)
  * 해당 호출이 **서버 사이드인지 클라이언트 사이드인지** (맥락 차이에 따른 동작 변화 가능성)

**3단계: re-export 및 래퍼 체인 추적**

* 수정 예정 함수가 **re-export, barrel export, 래퍼 함수**를 통해 다른 모듈에 노출되는 경로 추적
* 중간 래퍼 함수가 시그니처를 변환하거나 제한하는 경우 → **해당 래퍼도 함께 수정해야 할 것인지** 판단
* 단순 참조 할당(re-export)인 경우 → 시그니처 변경이 자동 전파되므로 별도 수정 불필요 예상 확인

**4단계: 영향받지 않을 영역 명시적 확인**

* 수정 대상 함수와 **같은 모듈에 속하지만 변경하지 않을** 함수들이 정상 동작을 유지할 것인지 확인
* 수정 대상 테이블/데이터에 접근하는 **다른 경로**(읽기 전용 등)가 영향받지 않을 것인지 확인
* **UI/프론트엔드 코드**에서 해당 함수를 직접 호출하는 경우, 동작 변화가 없을 것인지 확인

**5단계: 종합 사전 영향도 보고**

* 조사 결과를 표 형식으로 정리하여 사용자에게 보고:
  * **변경 예정 항목** (어디에서, 무엇을, 어떻게 변경할 예정)
  * **영향 없을 호출** (호출 위치, 미영향 사유)
  * **영향 없을 파일** (import는 하지만 해당 함수 미사용)
* 하위 호환성 보장 메커니즘 명시 (기본 매개변수, optional 타입 등)
* re-export 체인의 자동 전파 여부 명시

**주의사항**

* **추측 금지:** "아마 괜찮을 것이다"는 허용하지 않음 — 반드시 `grep_search`로 **실제 코드 확인**
* **전수 조사 원칙:** import하는 파일을 먼저 찾고, 그 중 실제 호출하는 파일만 필터링 — 누락 방지
* **맥락 구분:** 동일 함수라도 서버 사이드/클라이언트 사이드 호출 맥락에 따라 동작이 다를 수 있음 (예: RLS, 인증 세션)
* **간접 영향:** 함수 시그니처뿐 아니라 반환값, 에러 처리, 부수 효과(side effect) 변경 가능성도 점검
* **이 단계에서는 코드를 수정하지 않는다.** 분석과 보고만 수행한다.

**분기 조건**

* **잠재적 영향 발견 시:** 영향받을 호출 위치를 구체적으로 보고하고, 해결책 문서에 추가 수정 사항을 반영할 것을 권고
* **사전 점검 통과 시:** "수정해도 안전하다"고 판정, 코드 수정 단계(`/5apply-safe-fix`)로 진행 가능
