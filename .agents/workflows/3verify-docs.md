# 워크플로우 3/4: 작성 문서 검수 및 공식 문서 기반 업데이트

> 디버깅 파이프라인: `[1] 원인 분석` → `[2] 해결책 문서화` → **`[3] 문서 검수`** → `[4] 코드 수정`

**■ 개요 및 원칙**

* **목적:** 기존 작성 문서와 최신 공식 문서 대조를 통한 팩트 체크 및 정보 보완
* **수정 원칙:** 기존 기능 및 디자인 온전하게 보존, 시스템 정합성 유지, 필수 부분만 최소한으로 수정

**■ 진행 절차**

**1단계: 기존 문서 검수 대상 확인**

* **검수 대상:** 문제 해결 문서(원인, 해결책 등) 상세 검토
* **항목화:** 기술적 주장, 코드 패턴, API 사양, 파라미터 설명 등 세부 항목 정리

**2단계: 프로젝트 환경 일치 여부 확인**

* **환경 대조:** 문서 내용과 현재 환경(라이브러리/프레임워크 버전, 설정) 일치 여부 확인
* **직접 조회:** `package.json` 및 설정 파일을 확인하여 실제 사용 버전과 비교

**3단계: 최신 공식 문서 팩트 체크**

* **문서 열람:** `read_url_content` 또는 `view_content_chunk`를 통해 공식 문서를 직접 열람 및 대조
* **주요 검증 항목:**
  * API 엔드포인트, 파라미터(명칭, 타입, 필수/선택 여부)
  * 응답 구조 및 필드명
  * 지원 옵션 값 및 기본값
  * 에러 코드 및 의미
  * 버전별 주요 변경 사항(Breaking Changes)

**4단계: 문제점 식별 및 분류**

* **낡은 정보:** 이전 버전에만 해당하고 현재는 변경된 사양
* **논리적 비약:** 근거 없이 도출된 결론이나 해결책
* **환각 (Hallucination):** 실제 공식 문서에 존재하지 않는 허위 내용
* **누락된 엣지 케이스:** 문서에서 고려하지 못한 예외 상황

**5단계: `.docs` 문서 수정 및 보완**

* **업데이트:** 문제 발견 시 공식 문서 기반으로 `.docs/` 파일 정확하게 수정
* **필수 명시 사항:**
  * **수정 항목:** 구체적인 변경 내용
  * **수정 근거:** 참고한 공식 문서 내용 및 URL

**■ 주의사항**

* **교차 검증 필수:** 기존 문서를 무비판적으로 신뢰하지 않으며, 모든 기술적 주장은 공식 문서로 검증함
* **미확인 표기:** 검증 불가한 내용은 "미확인"으로 표기 후 추후 확인 대상으로 분류
* **검증 완료 표기:** 팩트 체크 결과 이상이 없는 경우에도 "검증 완료"를 명시하여 신뢰도 확보

**■ 분기 조건**

* **해결책 자체가 잘못된 경우:** 검증 중 해결책의 근본 방향이 틀렸다고 판단되면 → `/2document-solution` 재실행
* **원인 가설 자체가 틀린 경우:** 원인 분석부터 재수행 필요 → `/1analyze-root-cause` 재실행
