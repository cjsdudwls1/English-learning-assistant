# 워크플로우: 공식 문서 기반 체계적 리팩토링 계획 (설계도 작성)

> 리팩토링 파이프라인: **`[설계도 작성]`** → `[코드 구현]` → `[검증]`

**■ 개요 및 원칙**

* **목적:** 스파게티 코드 분석 결과를 바탕으로, 코드를 독립적인 모듈로 분리하기 위한 **'마이크로 설계도'**를 작성
* **핵심 원칙:** 코드를 짜기 전에 반드시 아래 4가지 항목에 대해 구체적으로 `.docs` 폴더에 문서화할 것
* **수정 원칙:** 기존 기능 및 디자인 온전하게 보존, 시스템 정합성 유지

**■ 전제 조건**

* 스파게티 코드 분석 문서(`.docs` 폴더 내 분석 결과)가 존재할 것
* 리팩토링 대상 파일 및 문제점이 사전에 식별되어 있을 것

**■ 진행 절차**

**1단계: 분석 문서 확인 및 리팩토링 대상 선정**

* 사용자가 언급한 `.docs` 폴더 내 분석 문서 정독
* 리팩토링 대상 파일의 현재 코드를 직접 열어 구조, 의존성, 호출 관계 파악
* 대상 코드의 책임(Responsibility)을 기능 단위로 분류

**2단계: 마이크로 설계도 작성 (`.docs` 폴더에 문서화)**

아래 **4가지 항목 모두** 빠짐없이 구체적으로 작성한다.

---

### 항목 1. 아키텍처 및 폴더 구조 명세

* **새로 생성될 파일 목록:** 쪼개져서 새로 생성될 파일들의 **정확한 이름과 디렉토리 경로**를 명시
  * 예시: `src/services/videoSync.ts`, `src/hooks/useCartValidation.ts` 등
* **적용 디자인 패턴:** 어떤 디자인 패턴이나 구조를 적용할 것인지 명시
  * 예시: Strategy 패턴, Custom Hook 분리, 팩토리 패턴, Barrel Export 패턴 등
* **폴더 트리 시각화:** 변경 전/후의 폴더 구조를 트리 형태로 비교 제시

---

### 항목 2. I/O(입출력) 계약 무결성 보장

* **인터페이스 정의:** 새로 쪼개진 모듈의 Input(파라미터/Props)과 Output(리턴값/상태)의 **데이터 타입(인터페이스)**을 어떻게 고정할 것인지 명시
  * TypeScript 인터페이스/타입 초안 포함
* **블랙박스화 검증:** 기존에 이 코드를 호출하던 **'다른 파일들'**이 전혀 수정 없이 그대로 작동할 수 있는지 검증
  * 기존 호출부 전수 조사 (grep/검색을 통해 호출부 목록 작성)
* **호출부 수정이 불가피한 경우:** 수정이 불가피하다면 그 **영향 범위를 어떻게 최소화**할 것인지 구체적으로 설명
  * 영향받는 파일 목록과 수정 내용 명시
  * re-export, adapter 패턴 등 호환성 유지 전략 제시

---

### 항목 3. 최신 공식 문서 레퍼런스

* **공식 문서 근거 제시:** 제안하는 구조가 현재 스택의 **최신 공식 문서에서 권장하는 패턴**이라는 것을 증명
  * 반드시 `read_url_content`, `context7`, 또는 `search_web` 도구를 활용하여 공식 문서를 **직접 확인**
* **참고 대상 스택:** 프로젝트에서 사용 중인 주요 기술 스택 기준
  * 예시: Next.js App Router, React 18+, Supabase, TypeScript 등
* **문서화 형식:** 각 참고 문서에 대해 아래 형식으로 명시
  * 참고 개념/패턴명
  * 공식 문서 URL 또는 섹션명
  * 해당 패턴이 현재 리팩토링에 어떻게 적용되는지 1~2줄 설명

---

### 항목 4. 단계별 구현 마일스톤

* **구현 순서 분할:** 코드를 한 번에 쏟아내지 않도록, **구현 순서를 쪼개서** 제시
  * 각 단계는 독립적으로 완료 가능하고, 단계별로 빌드/타입 체크가 통과해야 함
* **마일스톤 형식 예시:**
  * Step 1. 타입 인터페이스 정의
  * Step 2. 유틸 함수 분리
  * Step 3. Custom Hook 추출
  * Step 4. 메인 컴포넌트 결합
  * Step 5. 기존 호출부 연결 및 호환성 검증
* **각 단계별 포함 사항:**
  * 해당 단계에서 생성/수정되는 파일 목록
  * 완료 기준 (타입 체크 통과, 빌드 성공 등)
  * 예상 위험 요소 및 대응 방안

---

**3단계: 설계도 자체 검증**

* 설계도의 각 항목이 빠짐없이 작성되었는지 체크리스트로 확인
* 기존 기능 보존 여부 재확인 (호출부 영향도 최종 점검)
* 공식 문서 근거가 실제로 유효한지 재확인

**■ 주의사항**

* **코드 작성 금지:** 이 워크플로우에서는 오직 설계도(문서)만 작성한다.
* **직접 확인 원칙:** 공식 문서를 직접 읽고 숙지한 상태에서만 패턴을 제안한다.
* **기존 기능 보존:** 리팩토링으로 인해 기존 기능이 절대 망가지지 않도록 I/O 계약을 철저히 정의한다.
* **점진적 구현:** 마일스톤은 반드시 각 단계별로 독립 검증이 가능하도록 분할한다.
