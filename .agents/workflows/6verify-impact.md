# 워크플로우 6: 수정 후 영향도 검증 (Impact Analysis)

> 디버깅 파이프라인: `[1] 원인 분석` → `[2] 해결책 문서화` → `[3] 문서 검수` → `[4] 코드 수정` → **`[6] 영향도 검증`**

**■ 개요 및 원칙**

* **목적:** 코드 수정이 기존 정상 작동 기능·로직에 악영향(사이드 이펙트)을 미치지 않았는지 체계적으로 조사·검증
* **핵심 원칙:** "증거 기반 판단" — 추측이 아닌 코드 검색·타입 체크·빌드 결과로 안전성 입증

**■ 전제 조건**

* `/4apply-safe-fix` 또는 기타 코드 수정이 완료된 상태일 것
* 수정 대상 함수·모듈·파일이 명확히 식별된 상태일 것

**■ 진행 절차**

**1단계: 수정 범위 정리**

* 이번 수정에서 **변경된 함수 시그니처, 로직, 파라미터** 목록 정리
* 변경 유형 분류:
  * 시그니처 변경 (매개변수 추가/제거/타입 변경)
  * 내부 로직 변경 (분기, 반환값, 에러 처리)
  * import/export 경로 변경
* 변경된 항목마다 **하위 호환성 유지 여부** 판단 (기본값, optional 여부 등)

**2단계: 호출 체인 전수 조사**

* `grep_search`로 수정된 함수·모듈의 **모든 호출 위치** 식별
* 단순 import만 하고 해당 함수를 사용하지 않는 파일은 제외
* 각 호출 위치에 대해 다음을 확인:
  * 현재 호출 방식 (인자 수, 인자 타입)
  * 수정 후에도 **기존 호출이 그대로 동작하는지** (기본 매개변수, 타입 호환)
  * 해당 호출이 **서버 사이드인지 클라이언트 사이드인지** (맥락 차이에 따른 동작 변화 여부)

**3단계: re-export 및 래퍼 체인 추적**

* 수정된 함수가 **re-export, barrel export, 래퍼 함수**를 통해 다른 모듈에 노출되는 경로 추적
* 중간 래퍼 함수가 시그니처를 변환하거나 제한하는 경우 → **해당 래퍼도 수정 필요 여부 판단**
* 단순 참조 할당(re-export)인 경우 → 시그니처 변경이 자동 전파되므로 별도 수정 불필요 확인

**4단계: 영향받지 않는 영역 명시적 확인**

* 수정 대상 함수와 **같은 모듈에 속하지만 변경하지 않은** 함수들의 정상 여부 확인
* 수정 대상 테이블/데이터에 접근하는 **다른 경로**(읽기 전용 등)가 영향받지 않는지 확인
* **UI/프론트엔드 코드**에서 해당 함수를 직접 호출하는 경우의 동작 변화 여부 확인

**5단계: 타입 및 빌드 검증**

* **타입 체크:** 프로젝트의 정적 분석 도구로 시그니처 변경에 의한 타입 불일치 검출
  * TypeScript: `npx tsc --noEmit`
  * Python: `mypy`, `pyright` 등
  * Go: `go vet`
* **프로덕션 빌드:** 프로젝트의 빌드 명령으로 런타임 import 오류 및 빌드 실패 검출
  * Node.js: `npm run build` / `yarn build`
  * Python: `python -m py_compile` 또는 프레임워크별 빌드
  * Go: `go build ./...`
* 두 검증 모두 에러 없이 통과해야 함

**6단계: 종합 영향도 보고**

* 조사 결과를 표 형식으로 정리하여 사용자에게 보고:
  * **수정된 호출** (어디서, 무엇을, 어떻게 변경)
  * **영향 없는 호출** (호출 위치, 미영향 사유)
  * **영향 없는 파일** (import는 하지만 해당 함수 미사용)
* 하위 호환성 보장 메커니즘 명시 (기본 매개변수, optional 타입 등)
* re-export 체인의 자동 전파 여부 명시

**■ 주의사항**

* **추측 금지:** "아마 괜찮을 것이다"는 허용하지 않음 — 반드시 `grep_search`로 **실제 코드 확인**
* **전수 조사 원칙:** import하는 파일을 먼저 찾고, 그 중 실제 호출하는 파일만 필터링 — 누락 방지
* **맥락 구분:** 동일 함수라도 서버 사이드/클라이언트 사이드 호출 맥락에 따라 동작이 다를 수 있음 (예: RLS, 인증 세션)
* **간접 영향:** 함수 시그니처뿐 아니라 반환값, 에러 처리, 부수 효과(side effect) 변경도 점검

**■ 분기 조건**

* **영향 발견 시:** 영향받는 호출 위치를 구체적으로 보고하고, 추가 수정이 필요한 경우 `/4apply-safe-fix`로 돌아가 보완 수정 수행
* **검증 통과 시:** 수정 완료 확정, 사용자에게 안전성 보고
