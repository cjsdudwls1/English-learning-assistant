# 워크플로우 1/4: 근본 원인 분석 및 문서화

> 디버깅 파이프라인: **`[1] 원인 분석`** → `[2] 해결책 문서화` → `[3] 문서 검수` → `[4] 코드 수정`

**■ 개요 및 원칙**

* **목적:** 코드 에러, 버그 등 기술적 문제 해결 시, 코드 수정에 앞서 문제의 근본 원인을 정확히 파악하고 문서화
* **수정 원칙:** 기존 기능 및 디자인 온전하게 보존, 시스템 정합성 유지, 필수 부분만 최소한으로 수정

**■ 진행 절차**

**1단계: 에러 로그 및 동작 맥락 수집**

* **정보 수집:** 에러 로그, 스택 트레이스, 콘솔 출력, 브라우저 네트워크 탭 정보 등 누락 없이 확보
* **위치 특정:** 에러가 발생한 파일, 함수, 라인 번호 확인
* **흐름 추적:** 에러 발생 당시의 실행 흐름(호출 순서, 데이터 흐름 등) 파악

**2단계: 현재 코드의 동작 맥락 분석**

* **컨텍스트 분석:** 에러 발생 영역 및 주변 코드(관련 함수, 모듈, 의존성) 면밀히 검토
* **우선 분석:** 공식 문서 탐색이나 코드 수정 전, 수집된 로그와 현재 동작 맥락을 최우선으로 분석
* **이력 확인:** 최근 커밋, 배포 등 변경 사항과 문제의 연관성 파악

**3단계: 근본 원인 가설 수립**

* **심층 규명:** 표면적 오류(예: 500 에러, 타입 에러)를 넘어 "왜 발생했는지"에 대한 근본 가설 수립
* **다중 가설 정리:** 가능한 원인이 여러 개일 경우, 각 가설의 가능성과 판단 근거를 함께 명시
* **외부 요인 고려:** 환경(로컬/배포), 데이터 상태, 타이밍 등 비코드적 요인 포함

**4단계: `.docs` 폴더에 원인 문서화**

* **문서 작성:** 파악된 '원인 및 가설'에만 집중하여 프로젝트 내 `.docs` 폴더에 마크다운으로 명확히 기록
* **필수 포함 항목:**
  * **문제 현상 요약:** 발생한 에러와 위치
  * **에러 로그/스택 트레이스:** 핵심 로그 발췌
  * **분석 과정:** 코드 추적 순서 및 방법
  * **근본 원인 가설:** 문제 발생 사유 및 판단 근거

**■ 주의사항**

* **분석 집중:** 이 단계에서는 현상 파악, 원인 규명, 가설 수립에만 온전히 집중할 것
* **성급한 해결 지양:** 오진(誤診) 방지를 위해 충분한 분석 전에 해결책 제시 금지
* **명확한 파일명:** 문제 식별이 직관적으로 가능한 파일명 지정 (예: `video-generation-500-error-analysis.md`)
