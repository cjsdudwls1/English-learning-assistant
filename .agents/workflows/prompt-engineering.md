# 프롬프트 엔지니어링 워크플로우

사용자가 추가 기능 개발 요구사항을 자연어로 전달하면, 프로젝트 코드를 분석하여 AI 코딩 에이전트에게 전달할 수 있는 **컴팩트하고 정확한 프롬프트**를 생성한다.

## 절차

### 1. 관련 코드 분석
- 사용자의 요구사항에서 **수정 대상이 될 파일**을 식별한다.
- 주요 분석 대상:
  - 프론트엔드 페이지/컴포넌트 (예: `GenerateProblemsPage.tsx`)
  - 백엔드 Edge Function (예: `supabase/functions/generate-problems-by-type/index.ts`)
  - 타입 정의 (예: `src/types.ts`)
  - 프롬프트 템플릿 (예: `promptTemplates`, `buildPrompt()`)
- 파일의 **인터페이스, state 변수, 함수 시그니처, 기존 패턴**을 파악한다.

### 2. 프롬프트 작성 규칙
- **컴팩트하게**: 장황한 설명 없이 핵심만 기술.
- **파일 경로 명시**: 수정 대상 파일을 정확히 지정.
- **인터페이스 확장 명시**: 추가할 필드명, 타입을 구체적으로 지정.
- **프롬프트 주입 문구 직접 작성**: AI 모델에 전달될 실제 프롬프트 문장을 제공.
- **기존 패턴 유지 지시**: 다크모드, 반응형, 이중 언어(ko/en), 기존 코드 컨벤션 준수 강제.
- **조건부 로직 명확화**: 토글/모드에 따른 UI 표시 조건을 명시.

### 3. 프롬프트 구조 (각 기능별)
각 기능 항목은 아래 구조를 따른다:
```
■ N. 기능명
- UI 변경 사항 (어떤 컴포넌트에 무엇을 추가)
- 인터페이스 변경 (필드명?: 타입)
- 프롬프트 변경 (buildPrompt()에 어떤 문구를 주입)
- 생성 JSON 변경 (출력 필드 추가/변경)
```

### 4. 공통 규칙 섹션 포함
프롬프트 말미에 아래 공통 규칙을 항상 포함한다:
- 기존 코드 패턴(promptTemplates 구조, corsHeaders, model failover) 유지
- 다크모드/라이트모드, 반응형 레이아웃 패턴 준수
- 한국어/영어 이중 언어(language state 기반) 지원 유지

### 5. 결과물 저장
- artifacts 디렉토리에 `prompt_engineering.md`로 저장한다.
- 프롬프트 본문과 함께 **설계 포인트 요약 표**를 포함한다.
